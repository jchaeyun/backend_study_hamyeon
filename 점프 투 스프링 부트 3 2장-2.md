
## 1. Spring MVC: 컨트롤러 데이터 전달 방식의 재구성

Spring MVC에서 컨트롤러의 데이터를 뷰(View)로 전달하는 방법은 `Model` 객체 사용이 유일한 정답은 아니다. 설계 의도에 따라 다음과 같은 대안이 존재한다.

### 데이터 전달 방식의 종류
*   **Model 인터페이스:** `addAttribute`를 통해 데이터를 저장한다. 가장 범용적이다.
*   **ModelAndView:** 데이터와 뷰 이름을 하나의 객체로 관리한다. 반환 타입이 명확하다는 장점이 있다.
*   **Map / ModelMap:** 파라미터로 `Map`을 받아 데이터를 주입한다. 프레임워크 의존성을 낮추려는 의도로 사용된다.

### 기술적 판단 기준
*   **SSR(Server Side Rendering):** Thymeleaf와 같은 템플릿 엔진을 사용할 경우 위 방식 중 하나를 반드시 선택해야 한다.
*   **REST API:** 프론트엔드가 분리된 환경에서는 `@RestController`를 사용한다. 이 경우 모델 객체를 통하지 않고 DTO나 컬렉션을 직접 반환하며, 이는 JSON 형태로 직렬화된다.

## 2. Thymeleaf URL 표현식: 리터럴 치환 vs 표준 문법

타임리프에서 동적 URL을 생성할 때 사용하는 두 문법은 상호 보완적이다.

### 문법 비교
1.  **리터럴 치환 (`|...|`):** 문자열 합치기(`+`) 없이 변수와 텍스트를 혼합한다. 직관적이나 복잡한 쿼리 스트링 대응에는 한계가 있다.
2.  **표준 URL 표현식 (`@{...}`):** 타임리프의 권장 방식이다. 경로 변수(`{id}`)와 파라미터(`()`)를 명확히 분리한다.

### 실무적 권장사항
단순한 ID 매핑에는 `|...|`이 편리하지만, **URL 인코딩과 다중 파라미터 처리**가 필요한 환경에서는 표준 URL 표현식(`@{/path/{id}(id=${id}, type='new')}`)을 사용하는 것이 안전하다.

## 3. Java: int(Primitive)와 Integer(Wrapper)의 차이

자바의 자료형 선택은 메모리 구조와 데이터 상태 정의 방식에 직접적인 영향을 미친다.

### 핵심 차이점 요약

| 항목 | int (Primitive) | Integer (Wrapper) |
| :--- | :--- | :--- |
| **유형** | 기본 자료형 (값 자체 저장) | 객체 (참조 주소 저장) |
| **기본값** | `0` | `null` |
| **저장 공간** | Stack 메모리 | Heap 메모리 |
| **컬렉션 활용** | 사용 불가 | 사용 가능 (`List<Integer>`) |



### 주의사항: Autoboxing & NPE
Java 5 이후 두 타입 간 자동 변환(Autoboxing/Unboxing)이 지원되지만, `null` 상태인 `Integer`를 `int`로 변환할 때 **`NullPointerException`**이 발생할 위험이 있다. 따라서 데이터베이스와 연동되는 Entity 클래스 등 `null`이 발생할 수 있는 필드에는 `Integer`를 사용하는 것이 관례다.

## 4. Optional과 get() 메서드의 안전한 사용

`Optional<T>`은 존재하지 않을 수 있는 값을 다루기 위한 컨테이너다.

### 데이터 추출 패턴
*   **`get()`:** 값을 즉시 반환한다. 내부 값이 `null`일 경우 `NoSuchElementException`을 발생시키므로, 호출 전 반드시 `isPresent()`로 상태를 검증해야 한다.
*   **`orElseThrow()`:** 값이 없으면 예외를 명시적으로 던진다. 서비스 레이어에서 비즈니스 예외를 처리할 때 가장 권장되는 패턴이다.
*   **`orElse(null)`:** 예외 대신 `null`을 반환하여 후속 로직에서 분기 처리를 유도한다.
