
### 1. 깃(Git)과 깃허브(GitHub) 형상 관리
* **브랜치(Branch) 명칭 통합:** 로컬과 원격 저장소의 기준 브랜치는 반드시 `main`으로 통일함. 로컬이 `master`로 설정되어 충돌할 경우, `git branch -m master main`으로 이름을 변경하고 강제 푸시(`--force`)하여 깃허브를 덮어씀.
* **403 Forbidden (권한 에러):** 깃허브 접근 거부 시, 윈도우 '자격 증명 관리자'에 저장된 이전 계정의 캐시가 원인일 확률이 높음. 이를 지우거나, 깃허브 웹에서 **Personal Access Token (PAT)**을 발급받아 원격 주소(`remote origin`)에 직접 주입하는 것이 가장 빠르고 확실한 우회 경로임.
* **LF/CRLF 경고:** 윈도우(CRLF)와 리눅스(LF)의 줄바꿈 방식 차이로 인한 단순 경고임. 무시하고 푸시하거나, `git config --global core.autocrlf true` 명령어로 자동 변환을 설정함.

### 2. 스프링 부트(Spring Boot) 개발 원칙
* **패키지 대소문자 엄격 구분:** 로컬(Windows)과 달리 배포 서버(Linux)는 대소문자를 엄격히 구분함. 폴더명과 코드 내 `package`, `import` 선언부는 예외 없이 100% 소문자로 통일해야 `package does not exist` 컴파일 에러를 방지할 수 있음.
* **의존성(Dependency) 검증:** `build.gradle`에 존재하지 않는 패키지(예: `spring-boot-starter-validation-test`)를 명시하면 클라우드 빌드 환경에서 패키지를 찾지 못해 전체 프로세스가 중단됨. 테스트 도구는 `spring-boot-starter-test` 단일 의존성으로 해결함.
* **스케줄러 활성화:** DB 주기적 업데이트 등을 위한 `@Scheduled` 백그라운드 로직은, 프로젝트 최상단 메인 클래스에 `@EnableScheduling` 어노테이션이 선언되어 있어야만 작동함.
* **연쇄적 컴파일 에러:** 자바 컴파일러는 치명적 오류(오타, 패키지 누락 등) 발견 시 즉시 검사를 중단함. 하나의 오류를 해결하면 그제서야 다음 단계의 로직(의존성 주입 등) 오류가 화면에 나타나는 것이 정상적인 동작 방식임.

### 3. 환경 변수와 보안 (로컬 vs 배포 분리)
* **시크릿 데이터 하드코딩 금지:** DB 주소, 계정명, 비밀번호 등은 깃허브에 절대 올리지 않음. `application.yml`에 `${DB_URL}` 형태로 환경변수 치환 문법을 작성하고, 배포 플랫폼(Render) 대시보드에서 실제 값을 주입함.
* **로컬 실행 환경의 한계:** 환경변수 구조를 적용한 후 내 컴퓨터(IDE)에서 그냥 서버를 켜면 변수를 읽지 못해 DB 연결(`jdbcUrl`) 에러가 발생함. IDE의 실행 환경 설정(Edit Configurations)에 로컬 테스트용 환경변수를 직접 주입하거나, 실무 표준에 따라 로컬용 설정 파일(`application-local.yml`)과 배포용 설정 파일을 물리적으로 분리하여 사용함.

### 4. 도커(Docker) 및 클라우드(Render) 배포
* **Dockerfile의 존재 이유:** "내 자리에서는 되는데 서버에서는 안 되는" 환경 불일치 문제를 원천 차단하기 위함. 호스트 OS를 무겁게 복제하지 않고, 자바 실행에 필요한 라이브러리와 유저 스페이스만 가볍게 컨테이너로 격리하여 배포를 자동화함.
* **버전 동기화:** `build.gradle`의 Java 버전과 `Dockerfile` 베이스 이미지의 JDK 버전이 일치해야 함. 특정 소수점 버전(예: 8.15)보다는 범용 태그(`gradle:jdk17` 등)를 사용하여 버전 충돌 및 다운로드 에러를 방지함.
* **배포 에러 로그 역추적:** Render 배포 실패 시 맨 아래에 출력되는 `exit code: 1`은 결과 상태일 뿐임. 원인 파악을 위해서는 로그 스크롤을 위로 올려 `> Task :compileJava FAILED` 문구나 예외 스택(Stacktrace)이 출력된 지점을 찾아야 실제 문제(메모리 부족, 문법 오류, 버전 충돌 등)를 식별할 수 있음.


### 5. 도커(Docker) 컨테이너 내부 빌드 디버깅
* **로그 은폐 현상:** Render와 같은 클라우드 환경에서 Docker를 통해 빌드할 경우, 내부 프로세스(Gradle 등)에서 발생한 컴파일 에러의 상세 원인이 출력되지 않고 `exit code 1`이라는 단순 종료 상태만 로그에 남는 구조적 한계가 있음.
* **디버깅 모드 강제 주입:** 에러의 실체를 확인하기 위해서는 `Dockerfile` 내부의 빌드 명령어에 상세 로그 출력 옵션을 임시로 주입해야 함.
  * *수정:* `RUN gradle clean build -x test --no-daemon --info --stacktrace`
* **사후 처리:** 원인(예: 버전 충돌, 문법 오류)을 파악하고 해결한 뒤에는, 빌드 속도 최적화와 로그 용량 절감을 위해 해당 디버깅 옵션을 다시 제거하는 것이 원칙임.

### 6. 프론트엔드(FE)와의 연동 및 권한 설정 (가장 중요)
서버 배포가 완료되었다고 해서 프론트엔드가 즉시 API를 호출할 수 있는 것은 아님. 다음 2가지 사전 작업이 필수적임.
* **CORS (교차 출처 리소스 공유) 허용:** 브라우저는 기본적으로 서로 다른 도메인 간의 데이터 통신을 보안상 차단함. 백엔드 스프링 부트 설정(`WebConfig` 클래스의 `addCorsMappings` 메서드)에 프론트엔드의 도메인(예: `localhost:3000` 또는 실제 배포 주소)을 명시적으로 허용해야만 통신 오류(CORS Error)가 발생하지 않음.
* **API 명세서 및 베이스 URL 전달:** 프론트엔드 개발자에게 클라우드 서버의 기본 접속 주소(Base URL)와 함께, 각 기능별 HTTP 메서드(GET/POST 등), 엔드포인트 경로, 요청 헤더/바디 포맷, 예상되는 응답 JSON 구조를 문서화하여 전달해야 함.

### 7. 원격 저장소 충돌 및 강제 병합 (Git Push Rejected)
* **발생 원인:** 깃허브(Remote)에 로컬 컴퓨터(Local)에는 없는 파일(예: 레포지토리 생성 시 자동 추가된 `README.md` 등)이 존재할 때, Git은 데이터 유실을 막기 위해 푸시를 거부함.
* **해결 경로:**
  1. **로컬이 절대적 기준일 때:** 기존 깃허브의 내용을 무시하고 내 코드로 완전히 덮어씌워도 무방하다면 `git push --force`를 사용하여 강제 병합함. (초기 세팅 시 주로 사용)
  2. **안전한 병합이 필요할 때:** `git pull origin main --rebase` 명령어로 원격 저장소의 파일을 내 컴퓨터로 먼저 가져와 합친 뒤, 다시 푸시를 진행함.

### 8. 클라우드 무료 티어(Free Tier)의 물리적 한계 인지
* **빌드 지연 (Build Time):** 클라우드 환경에서의 첫 도커 빌드는 수백 MB의 자바 베이스 이미지를 다운로드하고(Image Pull), `build.gradle`에 명시된 수십 개의 의존성 라이브러리를 외부에서 새로 가져와야 하므로 최소 3분에서 최대 10분 이상 소요되는 것이 정상임.
* **콜드 스타트 (Cold Start):** Render 등 무료 호스팅 서비스는 일정 시간 동안 외부 요청(트래픽)이 없으면 서버 자원 회수를 위해 컨테이너를 수면(Spin-down) 상태로 전환함. 이후 최초의 요청이 들어오면 서버를 다시 깨우는 데 50초 이상의 지연 시간이 발생함. 이는 오류가 아닌 무료 인프라의 특성임.

### 9. 컨테이너의 무상태성(Stateless)과 데이터 영속성
* **구조적 한계:** Docker 컨테이너는 일회성(Ephemeral) 환경임. Render 서버가 절전 모드에 들어가거나 재시작되면 컨테이너 내부(예: `src/main/resources`)에 임시로 저장된 모든 로컬 데이터와 업로드된 파일은 영구적으로 삭제됨.
* **대안:** 데이터베이스는 외부 클라우드 DB에 분리하여 연결하고, 빈티지 상품 이미지 등 사용자가 업로드하는 정적 파일은 로컬 서버가 아닌 외부 객체 스토리지(AWS S3 등)로 전송되도록 아키텍처를 물리적으로 분리해야 함.

### 10. 빌드 시 테스트 생략(`-x test`)의 리스크
* **사실 관계:** 배포 에러 우회를 위해 `Dockerfile`에서 `gradle clean build -x test` 명령어를 사용하여 테스트 과정을 강제로 건너뛰었음. 이는 클라우드 환경에서 발생할 수 있는 DB 연결 실패로 인한 빌드 중단을 막아줌.
* **리스크:** 치명적인 비즈니스 로직 오류를 검증 없이 운영 환경(Production)까지 끌고 가는 위험이 발생함. 코드를 깃허브에 푸시하기 전 로컬 환경에서 단위 테스트를 검증하거나, GitHub Actions를 통해 배포 전 단계에서 테스트만 격리하여 실행하는 파이프라인 설계가 요구됨.

### 11. 동시성(Concurrency)과 트랜잭션 제어
* **구조적 한계:** 다수의 사용자가 동일한 상품에 동시에 접근하여 상태를 변경(예: 입찰 시도)할 경우, 데이터베이스에 경쟁 상태(Race Condition)가 발생하여 데이터 무결성이 훼손됨. 단순한 CRUD API만으로는 이를 방어할 수 없음.
* **대안:** DB 엔진 수준의 락(Pessimistic/Optimistic Lock) 메커니즘이나 트랜잭션 격리 수준(Isolation Level)을 백엔드 비즈니스 로직에 적용해야 함. 이는 OS의 뮤텍스(Mutex)나 세마포어와 동일한 저수준 시스템의 동시성 제어 원리가 웹 애플리케이션 계층에 적용되는 핵심 지점임.
